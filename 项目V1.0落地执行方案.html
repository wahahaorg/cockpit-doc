项目 V1.0 落地执行方案 ​

文档版本：V1.0

负责人：待填写
更新时间：2026-07-02

一、项目背景 ​

当前项目已完成老板需求对齐，并通过 AI 编码工具产出可交互前端 Demo，初步验证了产品方向和需求可行性。

但当前 Demo 仍处于静态原型阶段：

  • 所有展示内容为硬编码模拟数据；
  • 未接入数据库；
  • 无后端服务；
  • 无 RAG 知识库；
  • 无真实数据持久化能力；
  • 暂无标准化工程结构；
  • 暂无正式多人协作文档和接口契约。

本方案目标是将现有 Demo 从“可演示原型”推进为“可内测 V1.0 产品”。

二、V1.0 核心目标 ​

V1.0 的核心目标不是扩展大量新功能，而是完成产品从 Demo 到可运行系统的最小闭环。

2.1 V1.0 必须完成 ​

  1. 前端工程化改造完成，代码结构支持多人协作；
  2. 页面模拟数据全部抽离，具备统一数据管理方式；
  3. 前端请求层封装完成，预留后端接口接入位；
  4. 全页面字段需求清单梳理完成；
  5. 后端服务搭建完成；
  6. 核心业务数据可真实存储和读取；
  7. 前后端接口完成联调；
  8. 基础 RAG / AI 问答能力接入；
  9. 加载、空数据、错误、权限等异常状态具备兜底能力；
  10. 形成可演示、可测试、可持续迭代的 V1.0 内测版本。

2.2 V1.0 暂不追求 ​

  1. 不追求复杂权限体系；
  2. 不追求完整运营后台；
  3. 不追求大规模并发性能；
  4. 不追求复杂数据分析报表；
  5. 不追求全自动 AI Agent 工作流；
  6. 不追求多模型动态路由；
  7. 不追求复杂知识库权限隔离；
  8. 不追求一次性覆盖全部边缘场景。

2.3 V1.0 成功标准 ​

V1.0 版本需要满足以下标准：

  • 核心业务流程可以完整跑通；
  • 页面体验与原 Demo 保持一致或更稳定；
  • 核心数据来自真实接口和数据库；
  • 用户操作后数据可以持久化；
  • 基础 AI / RAG 能力可以回答核心场景问题；
  • 错误、空状态、加载中状态可被正确处理；
  • 版本可以部署到内测环境；
  • 团队可以基于该版本继续迭代。

三、当前项目状态 ​

3.1 已完成 ​

  • 老板需求初步对齐；
  • 产品方向通过 Demo 完成可视化表达；
  • 前端 Demo 已可交互；
  • 核心页面和基础交互已具备演示价值。

3.2 未完成 ​

  • 前端工程结构标准化；
  • 组件拆分与复用；
  • 硬编码数据抽离；
  • 接口请求层封装；
  • 页面字段需求清单；
  • 后端服务；
  • 数据库设计；
  • RAG 知识库；
  • 大模型调用链路；
  • 测试用例；
  • 部署方案；
  • 产品文档体系。

3.3 主要风险 ​

  1. Demo 页面字段不等于业务数据模型，后端设计不能完全被页面字段牵引；
  2. RAG 能力如果没有提前定义评测标准，后期容易反复返工；
  3. 前端工程化如果只做目录整理，不输出字段清单，会影响后端启动；
  4. 需求如果继续口头传递，容易造成接口和数据结构反复变化；
  5. 团队成员陆续入职，若缺少交接文档，会导致理解不一致。

四、团队角色与职责 ​

4.1 团队负责人 / AI 产品总负责人 ​

核心职责：

  • 明确 V1.0 产品边界；
  • 判断必做、暂缓、不做功能；
  • 确认核心业务流程；
  • 确认数据规则；
  • 确认 AI 能力边界；
  • 审核接口与数据模型关键决策；
  • 组织版本验收和老板演示。

负责人不介入具体编码，但必须参与以下关键节点：

  • 阶段 0 产品方向确认；
  • 阶段 1 需求边界确认；
  • 阶段 3 接口与数据模型评审；
  • 阶段 6 V1.0 版本验收。

4.2 AI 前端工程师 ​

当前阶段核心职责：

  1. 将现有 Demo 重构为标准前端工程；
  2. 搭建 Git 仓库和项目目录结构；
  3. 拆分页面、组件、状态和请求层；
  4. 抽离所有硬编码模拟数据；
  5. 封装统一请求方法；
  6. 预留后端接口地址；
  7. 梳理全页面数据字段需求；
  8. 补全加载、空数据、网络错误等兜底状态。

阶段交付物：

  • 标准化前端工程；
  • 模拟数据文件；
  • 请求方法封装；
  • 页面字段需求清单；
  • 前端接口字段需求表；
  • 异常状态页面。

4.3 AI 全栈工程师 ​

入职后核心职责：

  1. 结合前端字段需求和业务数据模型设计数据库；
  2. 搭建后端服务；
  3. 开发核心业务接口；
  4. 实现数据持久化；
  5. 完成前后端联调；
  6. 搭建基础 RAG 服务；
  7. 接入大模型调用；
  8. 配置部署环境和基础安全策略。

阶段交付物：

  • 后端工程；
  • 数据库表结构；
  • 接口契约文档；
  • API 服务；
  • RAG 服务初版；
  • 部署说明。

4.4 AI 产品经理 ​

入职后核心职责：

  1. 接管需求文档和迭代计划；
  2. 补全 PRD、流程说明、权限规则、字段说明；
  3. 统一接收团队需求疑问；
  4. 管控需求变更；
  5. 输出周报和版本汇报；
  6. 协助负责人组织评审和验收。

阶段交付物：

  • PRD；
  • 需求变更记录；
  • 迭代计划；
  • 版本验收记录；
  • 项目进度报告。

4.5 RAG / AI 效果评测工程师 ​

入职后核心职责：

  1. 构建真实场景测试问题集；
  2. 评测 RAG 召回准确率；
  3. 评测 AI 回答准确性、幻觉、拒答、引用来源；
  4. 输出异常案例；
  5. 推动 Prompt、分片、检索策略优化；
  6. 复测并出具 AI 效果验收报告。

阶段交付物：

  • 测试问题集；
  • AI 效果评测报告；
  • 异常案例清单；
  • 复测记录；
  • AI 效果验收报告。

五、分阶段执行计划 ​

阶段 0：产品方向与 V1.0 边界确认 ​

周期：0.5-1 天
负责人：团队负责人
参与人：AI 前端工程师，后续可补 AI 产品经理

目标：在工程改造前，先确认 V1.0 到底要解决什么问题。

需要确认：

  • 目标用户是谁；
  • 核心业务场景是什么；
  • 用户最重要的 1-2 个痛点是什么；
  • V1.0 必须跑通的核心流程是什么；
  • Demo 中哪些只是演示效果，不进入 V1.0；
  • AI 能力在流程中承担什么角色；
  • V1.0 成功标准是什么。

交付物：

  • 《V1.0 产品边界说明》
  • 《V1.0 必做 / 暂缓 / 不做清单》
  • 《V1.0 验收标准》

阶段 1：需求固化与 Demo 技术交底 ​

周期：1-3 天
负责人：团队负责人
参与人：AI 前端工程师

目标：把零散 Demo 和口头需求转化为团队可执行标准。

主要工作：

  1. 基于现有 Demo 和原始需求，生成需求边界说明；
  2. 梳理页面、功能、交互、字段；
  3. 标注现有硬编码数据位置；
  4. 标注待重构模块；
  5. 明确前端工程化交付标准；
  6. 与前端工程师完成任务交底。

交付物：

  • 《需求边界说明书》
  • 《功能与数据字段清单》
  • 《Demo 技术交底文档》
  • 《前端工程化改造任务书》

验收标准：

  • 前端工程师明确两周任务；
  • 每个页面需要的数据字段有初步记录；
  • 已明确哪些功能进入 V1.0；
  • 已明确哪些功能不进入 V1.0。

阶段 2：前端工程化改造 ​

周期：2 周
负责人：AI 前端工程师
参与人：团队负责人

目标：将不可协作的静态 Demo 改造成可对接后端、可多人协作的标准前端工程。

主要工作：

  1. 搭建 Git 仓库；
  2. 整理项目目录；
  3. 拆分公共组件；
  4. 抽离硬编码模拟数据；
  5. 统一管理 mock 数据；
  6. 封装请求方法；
  7. 预留接口地址；
  8. 增加加载状态；
  9. 增加空数据状态；
  10. 增加网络错误状态；
  11. 输出页面字段需求清单；
  12. 输出前端接口字段需求表。

交付物：

  • 标准化前端工程；
  • 组件目录结构；
  • mock 数据目录；
  • 请求封装模块；
  • 页面字段需求清单；
  • 前端接口字段需求表；
  • 异常状态页面。

验收标准：

  • 页面交互与原 Demo 一致；
  • 所有硬编码展示数据已抽离；
  • mock 数据可统一维护；
  • 接口接入位完整；
  • 字段清单可作为接口设计输入；
  • 异常状态可被模拟验证。

阶段 3：接口评审与数据模型确认 ​

周期：1-2 天
负责人：团队负责人 / AI 全栈工程师
参与人：AI 前端工程师，AI 产品经理

目标：基于前端字段需求，形成正式接口契约和数据模型，避免后端完全围绕页面字段建表。

主要工作：

  1. 审阅前端页面字段需求；
  2. 梳理业务实体；
  3. 确认数据库核心对象；
  4. 确认接口路径；
  5. 确认入参、返回、错误码；
  6. 确认权限规则；
  7. 确认分页、筛选、排序规则；
  8. 确认数据状态流转。

交付物：

  • 《接口契约文档》
  • 《数据模型说明》
  • 《权限与错误码说明》
  • 《前后端联调计划》

验收标准：

  • 前端知道如何接接口；
  • 后端知道如何建表和开发接口；
  • 负责人确认关键业务数据规则；
  • 接口进入短期冻结状态，非必要不变更。

阶段 4：后端与数据层开发 ​

周期：2 周
负责人：AI 全栈工程师
参与人：AI 前端工程师

目标：补齐后端服务和数据库能力，让核心业务流程使用真实数据运行。

主要工作：

  1. 搭建后端项目；
  2. 设计数据库表；
  3. 配置本地开发环境；
  4. 开发核心业务接口；
  5. 实现数据持久化；
  6. 加入基础错误处理；
  7. 加入基础日志；
  8. 输出接口文档；
  9. 与前端完成联调；
  10. 准备内测部署环境。

交付物：

  • 后端工程；
  • 数据库表结构；
  • API 接口；
  • 接口文档；
  • 联调版本；
  • 部署说明。

验收标准：

  • 前端核心页面可读取真实接口；
  • 核心操作可写入数据库；
  • 数据刷新后不丢失；
  • 接口错误可被前端正确处理；
  • 本地和内测环境可运行。

阶段 5：基础 RAG / AI 能力接入 ​

周期：1-2 周，可与阶段 4 后半段并行
负责人：AI 全栈工程师
参与人：RAG / AI 效果评测工程师

目标：完成基础知识库、检索和大模型调用链路，让 AI 能力进入真实业务流程。

主要工作：

  1. 明确知识来源；
  2. 整理知识文档；
  3. 设计文档分片策略；
  4. 接入向量数据库；
  5. 搭建检索服务；
  6. 接入大模型；
  7. 设计基础 Prompt；
  8. 定义回答格式；
  9. 定义拒答规则；
  10. 定义引用来源规则。

交付物：

  • 知识库初版；
  • RAG 服务；
  • Prompt 配置；
  • AI 回答规则；
  • AI 问答接口；
  • 初版 AI 问答能力。

验收标准：

  • 可以基于知识库回答核心问题；
  • 明显无依据的问题可以拒答；
  • 回答格式稳定；
  • 可以返回引用来源或依据说明；
  • 可以记录异常问题用于后续评测。

阶段 6：测试、评测与 V1.0 内测验收 ​

周期：1 周
负责人：团队负责人 / RAG 评测工程师 / 测试负责人
参与人：全体相关成员

目标：完成 V1.0 内测前验收，确认产品具备演示、试用和继续迭代条件。

主要工作：

  1. 核心业务流程测试；
  2. 前端异常状态测试；
  3. 接口稳定性测试；
  4. 数据持久化测试；
  5. 权限和安全基础检查；
  6. RAG 问答效果测试；
  7. 异常案例记录；
  8. Bug 修复；
  9. 复测；
  10. 准备版本演示材料。

交付物：

  • 测试用例；
  • Bug 清单；
  • 修复记录；
  • AI 效果评测报告；
  • V1.0 验收记录；
  • 内测演示版本。

验收标准：

  • 核心流程无阻断问题；
  • 关键页面无明显体验问题；
  • 数据可正常保存和读取；
  • AI 回答达到最低业务可用标准；
  • 版本可以在内测环境稳定运行；
  • 老板或业务方可完成版本演示和试用。

十二、项目验收口径 ​

V1.0 不以“功能多”为验收标准，而以“核心闭环真实可用”为验收标准。

最终验收需要回答：

  1. 用户是否能完成核心业务流程？
  2. 页面是否稳定可用？
  3. 数据是否来自真实接口？
  4. 操作结果是否可持久化？
  5. AI 是否能在核心场景中提供可用回答？
  6. AI 是否知道不能回答什么？
  7. 系统是否可以部署到内测环境？
  8. 团队是否可以基于该版本继续迭代？

只有以上问题得到明确确认，项目才算从 Demo 阶段进入 V1.0 内测阶段。
