# CEO 现金流驾驶舱 V1.0 财务数据接入与产品对接说明

文档状态：对接讨论稿  
目标读者：AI 产品经理、财务/CFO、技术负责人  
本文目标：明确为什么当前需要开展财务数据对接、产品经理需要与财务确认什么，以及确认完成后应交付哪些材料。

## 一、当前为什么需要开展数据对接

现有项目已经形成驾驶舱 Demo，并初步定义了现金、回款、支出、风险和待办等展示指标，但当前数据均为 Mock 数据。

要将 Demo 推进为真实可用的 V1.0，必须先解决这些指标依赖的原始数据从哪里来、如何进入系统、如何更新以及如何计算的问题。

当前文档中的“本月可用现金”“本月预计回款”“本月现金流缺口”等字段，属于驾驶舱展示指标或系统计算结果，不是 Excel 导入字段。完整链路应为：

```text
财务原始数据
→ 标准 Excel 模板
→ 上传、解析、校验和确认
→ 业务明细入库
→ 规则计算
→ 指标快照
→ 驾驶舱展示与 AI Tool 查询
```

如果不先明确数据接入规则，技术团队无法稳定完成：

- 标准 Excel 模板；
- 数据库表结构；
- Excel 解析和校验逻辑；
- 数据新增、更新和作废逻辑；
- 指标计算；
- 驾驶舱真实接口；
- AI 数据查询 Tool。

## 二、本次对接的目标

本次对接不是建设完整财务 ERP，也不是要求财务提供完整账务、凭证和总账数据。

对接目标是：

> 围绕 CEO 现金流经营研判，确定能够支撑 V1.0 核心指标的最小财务业务数据，并将其固化成由项目方提供的标准导入模板。

产品经理需要和财务共同确认：

1. 财务当前是否掌握所需数据；
2. 数据的真实业务含义和粒度；
3. 数据之间如何关联；
4. 哪些是原始事实，哪些由系统计算；
5. 数据如何上传、更新、修正和发布；
6. 财务是否能够持续按标准模板提供数据。

## 三、V1.0 驾驶舱问题与数据需求

| 驾驶舱问题 | 需要财务提供的数据 |
| --- | --- |
| 现在有多少钱 | 账户余额快照 |
| 本月预计进多少钱 | 应收款、预计到账日期 |
| 本月已经进多少钱 | 实际回款记录 |
| 本月准备花多少钱 | 计划支出 |
| 管理费用还能花多少 | 预算及已占用金额 |
| 哪些款已经逾期 | 应收款、到期日、实际回款 |
| 谁需要处理 | 客户/业务与责任人关系 |

V1.0 建议优先确认以下四类数据：

1. 现金余额；
2. 应收款；
3. 实际回款；
4. 计划支出。

预算和客户责任人关系可以同步确认；如果首期时间有限，可根据驾驶舱实际范围分批落地。

## 四、产品经理需要与财务确认的内容

### 4.1 现有数据情况

- 财务目前有哪些 Excel 可以提供上述数据？
- 是否可以提供脱敏样表和样例数据？
- 每张表由谁维护、谁复核？
- 数据多久更新一次？
- 是否存在同一类数据由多人维护、格式不一致的情况？

现有财务 Excel 只用于理解业务事实和数据现状。V1.0 最终应由项目方输出标准模板，不长期兼容各种历史表格格式。

### 4.2 每张表的业务含义

需要逐表确认：

- 这张表解决什么业务问题？
- 每个 Sheet 的用途是什么？
- 一行数据代表什么：账户快照、应收事项、实际到账事件还是支出计划？
- 表中哪些是明细行，哪些是汇总行？
- 哪些字段由人工填写，哪些是 Excel 公式？

### 4.3 字段定义

每个字段至少需要确认：

- 中文名称；
- 业务定义；
- 示例值；
- 是否必填；
- 数据类型；
- 金额单位和币种；
- 日期口径；
- 允许值范围；
- 是否允许为空；
- 是原始事实还是计算结果。

例如，“应收金额”必须确认是合同总金额、当期应收金额，还是当前未回余额。

### 4.4 唯一标识

每类数据必须能唯一识别，否则系统无法判断重复、新增和更新。

需要确认：

- 是否存在账户编号；
- 是否存在客户编号；
- 是否存在合同编号；
- 是否存在应收单号；
- 是否存在回款流水号；
- 是否存在支出计划编号；
- 如果当前没有唯一编号，财务能否在标准模板中补充？

### 4.5 数据之间的关联

需要确认主要业务关系：

```text
客户 → 合同/业务 → 应收事项 → 实际回款
部门/项目 → 预算 → 计划支出
客户/业务 → 责任人
```

重点问题包括：

- 客户、合同、应收和回款通过什么编号关联？
- 一个合同是否包含多笔应收？
- 一笔应收是否允许分多次到账？
- 一笔回款是否可能对应多笔应收？
- 客户重名或名称变化时如何识别？
- 责任人变更后，历史记录是否保留原责任人？

### 4.6 数据导入方式

每种模板都需要单独确认：

- 每次上传是全量快照还是增量追加？
- 同一个文件能否重复上传？
- 已存在记录是忽略、报错还是更新？
- 新文件缺少旧记录时，是否代表旧记录作废？
- 数据属于哪个业务日期和账期？
- 补录历史数据后是否需要重算历史指标？

不同数据类型的推荐方式如下：

| 数据类型 | 推荐导入方式 | 唯一键示例 |
| --- | --- | --- |
| 现金余额 | 按日期追加；同账户同日期更新 | 账户编号 + 统计日期 |
| 应收款 | 按应收单号新增或更新 | 应收单号 |
| 实际回款 | 增量追加；重复流水号拒绝或忽略 | 回款流水号 |
| 计划支出 | 按计划编号新增或更新 | 支出计划编号 |
| 预算 | 按周期和预算维度更新 | 周期 + 部门/项目 + 科目 |
| 客户与责任人 | 按客户编号新增或更新 | 客户编号 |

以上是技术推荐默认值，最终需要产品和财务结合实际业务确认。

### 4.7 字段修改规则

需要确认：

- 哪些字段允许更新？
- 哪些数据确认后不能直接修改？
- 空单元格表示“不修改”，还是“清空原值”？
- 金额变化是直接覆盖，还是保留调整记录？
- “已回款”等状态是否允许改回未回款？
- 已发布数据发现错误时如何更正？

空值语义必须明确，避免误清空正式数据。

### 4.8 删除、作废和历史保留

建议不通过 Excel 中“缺少某行”来删除系统数据，而是采用明确的有效/作废状态。

需要确认：

- 已发布数据能否删除？
- 应收取消后是删除还是作废？
- 错误回款如何冲销或更正？
- 已经参与历史指标计算的数据如何处理？
- 是否需要保留修改前后的版本？

V1.0 推荐：正式数据不物理删除，使用作废或更正，并保留追踪记录。

### 4.9 校验规则

产品和财务需要区分“阻止发布的错误”和“允许人工确认的警告”。

建议确认：

- 必填字段缺失；
- 编号重复；
- 客户或责任人不存在；
- 金额格式错误或小于零；
- 已回金额超过应收金额；
- 日期先后矛盾；
- 账期不匹配；
- 状态与金额矛盾；
- 跨表关联失败。

### 4.10 数据确认与发布

上传不等于正式入库。需要确认：

- 谁负责核对解析结果？
- 什么条件下数据可以发布？
- 是否允许只发布部分正确行？
- 发布失败是否整批回滚？
- 错误批次如何作废？
- 新数据发布后是否立即刷新驾驶舱？

V1.0 推荐采用：

```text
上传 → 解析 → 校验 → 人工确认 → 整批发布 → 指标重算
```

一个批次整体成功或整体失败，避免驾驶舱读取到半批数据。

## 五、原始数据与系统计算的边界

财务负责提供原始业务事实，不应人工填写系统可以确定性计算的结果。

### 财务提供

- 现金余额；
- 应收金额；
- 实际回款金额；
- 约定到账日期；
- 预计到账日期；
- 计划支出；
- 预算金额；
- 客户、业务和责任人关系。

### 系统计算

- 未回金额；
- 逾期天数；
- 回款风险等级；
- 本月预计回款；
- 本月实际回款；
- 本月计划支出；
- 管理费用可用额度；
- 现金流缺口；
- 今日待办和升级状态。

系统计算公式、阈值和口径需要由财务/CFO 确认，技术团队负责实现和版本管理。

## 六、标准模板形成路径

建议按以下流程推进：

```text
财务提供现有脱敏样表和业务说明
→ 产品确认 V1.0 需要哪些数据
→ 技术抽象业务实体、标准字段和关联关系
→ 项目方发布标准模板初稿
→ 财务试填
→ 产品、财务和技术联合评审
→ 修订并冻结模板 V1
```

标准模板应由项目方提供，财务按模板填写。现有财务表不是长期接口，只是设计标准模板的参考输入。

## 七、职责分工

### 产品经理负责

- 确认 V1.0 要回答的经营问题；
- 组织财务访谈；
- 梳理业务流程和异常处理方式；
- 确认哪些数据和规则进入 V1.0；
- 形成业务待确认项并推动闭环；
- 组织模板试填和联合评审。

### 财务/CFO 负责

- 提供现有脱敏样表；
- 解释字段业务含义和数据粒度；
- 确认所需数据能否持续提供；
- 确认唯一标识和业务关联依据；
- 确认计算公式、阈值和复核规则；
- 按标准模板试填并反馈；
- 最终确认模板和财务口径。

### 技术负责人负责

- 将业务要求转换为数据模型；
- 设计标准模板结构和字段规范；
- 设计唯一键、数据库约束和关联关系；
- 设计增量、更新、作废和版本策略；
- 设计解析、校验、确认和发布流程；
- 设计指标计算、API 和 AI Tool；
- 明确技术风险及无法实现的业务要求。

5. 《数据导入、更新与发布规则》；

8. 《待确认问题及责任人清单》；

## 九、完成标准

以下问题均得到明确答案后，本轮数据对接才算完成：

- [ ] V1.0 需要接入哪些类型的数据？
- [ ] 每种数据由哪个标准模板承载？
- [ ] 每张表的一行代表什么？
- [ ] 每个字段的业务含义和格式是什么？
- [ ] 每类数据的唯一标识是什么？
- [ ] 不同数据之间如何关联？
- [ ] 每种模板是全量、快照还是增量导入？
- [ ] 重复、修改、空值和作废如何处理？
- [ ] 哪些错误阻止发布，哪些仅提示？
- [ ] 谁负责核对和确认发布？
- [ ] 哪些字段由财务提供，哪些由系统计算？
- [ ] 数据变化会触发哪些指标重算？
- [ ] 财务已使用标准模板完成至少一次试填。

## 十、关键结论

本项目的数据接入不是对财务历史 Excel 做无限适配，而是基于实际业务和驾驶舱目标，由项目方制定标准模板。

产品经理本轮对接的核心任务不是收集若干 Excel 文件，而是推动财务事实、字段口径、关联关系和数据更新规则形成可确认、可执行的标准。

技术团队将在这些标准确认后，完成数据库表结构、Excel 解析、数据校验、指标计算、驾驶舱接口和 AI Tool 的正式设计。
