30 / 60 / 90 天现金流预测模块 MVP 技术方案初稿 ​

文档版本：V0.1
状态：讨论稿，计算口径待产品与财务确认
目标读者：产品、财务、后端开发和测试人员

1. 模块目标 ​

本模块基于已发布的财务基础数据，按天计算未来 90 天的预测现金余额，并回答以下问题：

  • 未来 30、60、90 天内是否会出现现金缺口；
  • 每个时间窗口内的最大现金缺口是多少；
  • 最大缺口最早在哪一天出现；
  • 现金何时跌破或恢复现金安全线；
  • 哪些预计回款和计划支出是缺口的主要影响因素。

MVP 中所有金额和日期由确定性规则计算。AI 不参与数值预测，只能在计算完成后解释原因和组织文字。

2. MVP 范围 ​

2.1 本期实现 ​

  • 生成从计算基准日开始、未来 90 天的每日现金余额序列；
  • 计算 30、60、90 天三个累计窗口的最大缺口；
  • 返回最低余额、缺口日期、首次跌破安全线日期和恢复日期；
  • 保留参与计算的应收和支出记录，支持结果追溯。

2.2 本期不实现 ​

  • 使用大模型生成回款金额或支出金额；
  • 根据历史规律自动预测没有计划日期的数据；
  • 概率预测、置信区间和多情景模拟；
  • 融资额度、授信、税费、汇率和集团内部资金调拨；
  • 复杂预算控制和会计利润预测。

3. 数据输入 ​

MVP 使用四类基础事实数据。

数据用途关键字段
账户余额确定预测起点的可用现金账户、快照日期、可用余额、状态
应收款确定未来可能到账的金额和日期应收金额、预计到账日、约定到账日、状态
实际回款计算每笔应收尚未到账的余额关联应收、回款日期、回款金额、状态
计划支出确定未来计划流出的金额和日期计划日期、计划金额、审批状态、状态

3.1 计算基准日 ​

每次计算必须指定 asOfDate，表示“站在哪一天看未来”。本文记为 D。

所有输入数据必须来自同一个已发布数据批次。未发布、已作废或校验失败的数据不得参与正式计算。

3.2 期初可用现金 ​

对每个有效账户，取不晚于 D 的最新一条余额快照：

text
账户期初余额 = 该账户 snapshot_date <= D 的最新 available_balance
期初可用现金 = Σ 所有有效账户的账户期初余额
1
2

如果任何有效账户在 D 之前没有余额快照，结果必须返回数据缺失警告。

3.3 应收未回金额 ​

先按应收款汇总有效实际回款：

text
累计实际回款 = Σ 关联到该应收的有效回款金额
应收未回金额 = max(应收金额 - 累计实际回款, 0)
1
2

累计实际回款大于应收金额时，未回金额仍按 0 处理，同时返回“可能存在超额回款”的数据警告。

3.4 有效预计回款日期 ​

MVP 的日期选择规则：

text
如果 expected_date 存在：有效预计回款日期 = expected_date
否则：有效预计回款日期 = agreed_due_date
1
2

只有同时满足以下条件的应收未回金额才进入基础预测：

  • 应收记录有效且处于未关闭状态；
  • 应收未回金额大于 0；
  • 有效预计回款日期位于 [D, D + 89 天]；
  • 有效预计回款日期没有早于计算基准日。

已逾期且没有新的未来预计到账日期的应收，不得假设在今天或明天到账。该金额单独标记为“逾期待确认回款”，不进入基础预测现金流。

3.5 有效计划支出 ​

计划支出同时满足以下条件时进入预测：

  • 记录有效；
  • 计划日期位于 [D, D + 89 天]；
  • 审批状态属于 planned、pending 或 approved；
  • 未被标记为已支付或已取消。

MVP 当前没有独立的实际付款表，因此系统无法通过真实付款记录自动扣除已经支付的计划。数据提供方必须及时更新计划支出状态，否则可能重复计算支出。这一限制必须随结果返回。

4. 逐日现金流计算 ​

4.1 计算日期范围 ​

生成 90 个自然日的计算点：

text
D, D+1, D+2, ..., D+89
1

4.2 每日流入 ​

text
第 t 日预计流入
= Σ 有效预计回款日期等于第 t 日的应收未回金额
1
2

4.3 每日流出 ​

text
第 t 日计划流出
= Σ 计划日期等于第 t 日的有效计划支出金额
1
2

4.4 每日净现金流 ​

text
第 t 日净现金流 = 第 t 日预计流入 - 第 t 日计划流出
1

4.5 每日预测余额 ​

text
第 0 日预测余额 = 期初可用现金 + 第 0 日净现金流
第 t 日预测余额 = 第 t-1 日预测余额 + 第 t 日净现金流
1
2

同一天发生多笔回款和支出时，MVP 按当日净额计算，不区分日内发生顺序。

5. 30 / 60 / 90 天窗口定义 ​

三个窗口均从 D 开始，属于累计窗口：

窗口日期范围
30 天D 至 D+29
60 天D 至 D+59
90 天D 至 D+89

页面展示的是窗口内曾经出现过的最大缺口，不是第 30、60、90 天当天的余额。

对任一窗口 W：

text
窗口最低余额 = min(窗口内每日预测余额)
窗口最大缺口 = max(现金安全线 - 窗口最低余额, 0)
缺口出现日期 = 最早达到窗口最低余额的日期
1
2
3

MVP 默认现金安全线为 0 元：

text
窗口最大缺口 = max(-窗口最低余额, 0)
1

采用窗口内最低余额，可以发现中途资金断点。例如第 20 天余额为负、第 30 天因回款恢复为正时，30 天窗口仍会正确显示曾经出现的缺口。

5.1 企业现金流规划中的产品口径 ​

企业通常不是分别建立“30 天模型”“60 天模型”和“90 天模型”，而是建立一条持续更新的滚动现金流预测，再从同一条预测曲线观察不同时间窗口。

本模块也采用这一方式：

text
四类基础事实数据
→ 生成未来 90 天逐日预测余额
→ 从同一条余额曲线截取 30、60、90 天累计窗口
→ 计算各窗口最低余额和最大缺口
1
2
3
4

三个窗口存在包含关系：60 天包含前 30 天，90 天包含前 60 天。因此三个缺口数字可能相同，也可能随着远期支出加入而逐步增大。它们不是三个互相独立的预测结论。

同时，预测距离越远，数据可靠性通常越低：

  • 未来 30 天的回款和支出计划相对明确，可作为短期资金调度依据；
  • 31～60 天主要用于观察已知计划带来的资金压力；
  • 61～90 天更适合判断趋势，不宜表达为与 30 天结果同等精确的承诺。

5.2 首页展示建议 ​

从老板决策视角看，同时突出三个“缺口金额”不一定是最容易理解的表达。MVP 后端仍应完整返回 30、60、90 天窗口结果，便于核对和后续扩展；首页建议优先突出：

  • 未来 90 天最大现金缺口；
  • 最大缺口出现日期；
  • 预计恢复安全线日期；
  • 一条未来 90 天预测余额曲线。

如果产品必须保留三张窗口卡片，建议增加口径标签：

窗口建议表达
30 天高可信短期预测
60 天计划预测
90 天趋势判断

最终页面采用“三个窗口缺口”还是“最大缺口 + 日期 + 恢复时间”，属于产品展示决策，不改变底层逐日计算逻辑。

6. 安全线与恢复日期 ​

MVP 默认：

text
现金安全线 = 0 元
1

派生日期定义如下：

text
首次跌破安全线日期
= 预测余额第一次小于现金安全线的日期

恢复安全线日期
= 跌破后预测余额第一次重新大于等于现金安全线，且此后不再跌破的日期
1
2
3
4
5

如果 90 天内没有恢复，返回空值并标记“90 天内未恢复”。后续版本可以把安全线升级为固定金额或未来若干天的刚性支出，但不进入当前 MVP。

7. 主要影响因素 ​

主要影响因素必须来自参与计算的真实记录，不由 AI 编造。

MVP 可按以下方式筛选：

  • 缺口日期之前未按计划到账的逾期应收，按未回金额降序；
  • 缺口日期当日及之前的计划支出，按计划金额降序；
  • 分别最多返回 3 笔预计回款和 3 笔计划支出。

计算引擎返回结构化记录及金额，AI 可以据此生成“主要原因”和“建议动作”的自然语言说明，但不得修改金额、日期和关联记录。

9. 输出结果最低要求 ​

计算结果至少包含：

  • 计算基准日和 90 天日期范围；
  • 数据批次和规则版本；
  • 期初可用现金；
  • 30、60、90 天窗口的最低余额、最大缺口和缺口日期；
  • 首次跌破安全线日期和恢复安全线日期；
  • 90 天每日预计流入、计划流出、净现金流和预测余额；
  • 主要影响因素及其来源记录；
  • 未进入预测的逾期待确认回款金额；
  • 数据缺失、超额回款和支出状态不完整等警告。

金额必须使用十进制定点计算，禁止使用二进制浮点数直接计算财务金额。

10. MVP 验收标准 ​

  • 相同数据批次、基准日和规则版本必须得到完全相同的结果；
  • 任一窗口的最大缺口能够追溯到每日余额和具体业务记录；
  • 已实际回款金额不会再次作为未来回款重复计入；
  • 已逾期且没有新预计日期的应收不会被自动假设到账；
  • 已取消的计划支出不会进入预测；
  • 30、60、90 天结果取各自窗口内最低余额，而不是窗口最后一天余额；
  • 财务可以使用同一批样本数据手工复算并得到一致结果。

11. 待确认口径 ​

以下事项不阻塞初稿，但在上线真实数据前必须确认：

  1. expected_date 是否允许业务人员持续更新；
  2. 已逾期应收是否全部排除，还是允许应用统一可信折扣；
  3. 计划支出哪些审批状态应进入预测；
  4. 实际付款数据缺失时，如何确认计划支出已经支付；
  5. 安全线是否长期使用 0 元，还是改为财务配置金额；
  6. 同一天资金流入和流出是否需要考虑日内先后顺序；
  7. 首页最终采用三个窗口缺口卡片，还是“最大缺口、出现日期、恢复日期”的决策型表达。
