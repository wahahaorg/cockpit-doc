# 付款建议 Top 3 模块 MVP 技术方案初稿

文档版本：V0.1  
状态：讨论稿，规则阈值和资料要求待产品与财务确认  
目标读者：产品、财务、后端开发和测试人员

## 1. 模块目标

本模块针对计划支出生成可追溯的付款建议，回答以下问题：

- 哪些计划支出目前具备付款资格；
- 多笔支出同时存在时应按什么顺序评估；
- 每笔支出支付后是否产生或扩大现金缺口；
- 系统建议可付、暂缓、补充依据，还是升级老板拍板；
- 建议产生后，AI 如何按需生成老板可读的原因和可选方案。

付款资格、排序、现金流试算和建议结果全部由确定性规则产生。AI 只解释已经计算出的结果，不决定是否付款。

## 2. MVP 范围

### 2.1 本期实现

- 使用计划支出现有字段完成基础资格检查；
- 对具备付款资格的支出生成稳定的付款队列；
- 按队列顺序逐笔进行未来 90 天现金流试算；
- 输出 `可付`、`暂缓`、`补充依据`、`老板拍板` 四类建议；
- 保存付款评估结果、原因代码和证据快照；
- 需要老板决策时生成老板拍板事件；
- 用户点击后，AI 根据结构化结果生成自然语言解释。

### 2.2 本期不实现

- 真实付款、银行指令或自动审批；
- 由 AI 判断资料是否合规或支出是否刚性；
- 完整合同、发票、结算单和验收单管理；
- 预算占用、供应商信用和采购合规；
- 自动融资或资金调拨。

## 3. 总体处理流程

```text
计划支出
→ 付款资格检查器
→ 付款优先级排序器
→ 现金流试算器
→ 建议决策器
→ 付款评估结果
→ 必要时生成老板拍板事件
→ 用户点击后调用 AI 解释器
```

其中前四个组件均为确定性代码。建议决策器可以先作为付款建议引擎中的纯规则函数，不要求拆成独立服务。

## 4. 数据输入

### 4.1 计划支出

主要使用：

- 支出编号和名称；
- 支出类别；
- 计划支付日期；
- 计划金额；
- 责任人；
- 刚性程度；
- 审批状态；
- 数据状态和数据批次。

### 4.2 现金流预测结果

付款试算复用 30 / 60 / 90 天现金流模块产生的未来 90 天逐日预测数据，包括：

- 期初可用现金；
- 每日预计回款；
- 每日计划支出；
- 每日预测余额；
- 当前最大缺口、缺口日期和安全线。

付款建议不得维护另一套独立现金流公式。

## 5. 付款资格检查器

### 5.1 现有字段检查

MVP 仅检查当前已有字段：

| 检查项 | 条件 | 资格结果 |
| --- | --- | --- |
| 金额 | 金额缺失或小于等于 0 | 补充/修正数据 |
| 计划日期 | 日期缺失 | 补充依据 |
| 支出类别 | 类别缺失 | 补充依据 |
| 责任人 | 责任人编号或名称缺失 | 补充依据 |
| 刚性程度 | 不是 `rigid` 或 `deferrable` | 补充依据 |
| 审批状态 | `planned` 或 `pending` | 尚未具备付款资格 |
| 审批状态 | `approved` | 进入付款排序和试算 |
| 审批状态 | `cancelled` | 排除，不生成付款建议 |

资料不完整时使用“补充依据”或“修正数据”，不直接输出“拒绝”。拒绝属于人工审批结论，不由数据完整性规则代替。

### 5.2 结构化结果

资格检查至少返回：

```json
{
  "eligible": false,
  "result": "needs_evidence",
  "reasonCodes": ["owner_missing"],
  "missingFields": ["owner_code"]
}
```

后续增加合同、结算单、验收单等字段时，只增加新的检查规则和原因代码，不改变后续排序、试算和决策流程。

## 6. 付款优先级排序器

审批状态主要负责资格分流。进入付款队列的正常情况下都是 `approved`，因此不再使用审批状态作为主要排序条件。

MVP 排序键依次为：

```text
1. 刚性程度：rigid 优先于 deferrable
2. 已超过计划日期的支出优先
3. 计划付款日期：日期更早的优先
4. 条件仍相同：按支出编号升序，保证结果稳定
```

金额不作为默认优先级。金额较小并不代表经营上更应该先支付，金额只参与后续现金流试算。

排序结果必须保存 `queue_order`，用于解释多笔付款之间的先后影响。

## 7. 现金流试算器

### 7.1 避免计划支出重复扣除

计划支出已经包含在 90 天现金流预测中。付款试算不能在原曲线上再次直接扣除，否则会重复计算。

首先构造试算基础曲线：

```text
试算基础曲线
= 原始 90 天现金流曲线
+ 所有待评估且已审批支出的金额
```

这里的“加回”表示从原预测中临时移除待评估支出。`planned` 和 `pending` 支出继续保留在预测中，作为尚未审批但已知的资金占用；`cancelled` 支出不参与预测。

### 7.2 逐笔试算

按 `queue_order` 将已审批支出逐笔重新加入基础曲线：

```text
候选支付后余额
= 当前已接受方案的每日余额
- 该笔支出在计划支付日的金额
```

每笔至少计算：

```text
支付前最低余额
支付后最低余额
支付前最大缺口
支付后最大缺口
缺口增加金额
是否首次跌破安全线
缺口出现日期
恢复安全线日期
```

```text
缺口增加金额 = max(支付后最大缺口 - 支付前最大缺口, 0)
```

### 7.3 是否加入后续试算曲线

- 支付后不产生或扩大缺口：加入当前方案，后续支出基于更新后的曲线试算；
- 可延后支出产生或扩大缺口：不加入当前方案，标记建议暂缓；
- 刚性支出产生或扩大缺口：按必须支付加入当前方案，同时升级老板拍板；
- 资格不通过的支出：不进入逐笔试算。

刚性支出即使导致缺口也不能被系统简单判定“不付”。系统应暴露资金缺口，并要求老板选择催收、调整其他支出或安排资金。

## 8. 建议决策器

建议映射规则：

| 条件 | 建议代码 | 页面文案 |
| --- | --- | --- |
| 资格不通过或资料不完整 | `needs_evidence` | 补充依据 |
| 审批尚未完成 | `not_ready` | 尚未具备付款资格 |
| 资格通过且支付后不产生或扩大缺口 | `pay` | 可付 |
| 可延后支出导致缺口 | `defer` | 建议暂缓 |
| 刚性支出导致缺口 | `boss_review` | 需要老板拍板 |

已取消支出在资格检查阶段直接排除，不写入付款评估结果。

判断标准使用计算结果和原因代码，不使用 AI 输出。

## 9. 付款建议 Top 3 展示排序

付款队列顺序用于试算，首页 Top 3 顺序用于突出最需要关注的建议，两者不能混为一谈。

首页建议按以下顺序展示：

```text
1. 需要老板拍板 boss_review
2. 建议暂缓 defer
3. 补充依据 needs_evidence
4. 尚未具备付款资格 not_ready
5. 可付 pay
```

同一建议类型内：

```text
缺口增加金额更大的优先
→ 计划日期更早的优先
→ 支出编号升序
```

取前三笔作为首页 Top 3。候选不足三笔时按实际数量展示，不使用模拟数据补足。

## 10. 付款评估结果表

建议新增 `payment_assessment`，用于保存规则引擎对每笔计划支出的评估结果。

建议字段：

| 字段 | 含义 |
| --- | --- |
| `id` | 主键 |
| `planned_expense_id` | 关联计划支出 |
| `as_of_date` | 计算基准日 |
| `import_batch_id` | 数据批次 |
| `rule_version` | 规则版本 |
| `queue_order` | 付款试算顺序 |
| `eligibility_result` | 资格检查结果 |
| `decision` | pay/defer/needs_evidence/not_ready/boss_review |
| `reason_codes` | 命中的结构化原因代码 |
| `min_balance_before` | 支付前最低余额 |
| `min_balance_after` | 支付后最低余额 |
| `gap_before` | 支付前最大缺口 |
| `gap_after` | 支付后最大缺口 |
| `gap_increase` | 缺口增加金额 |
| `gap_date` | 缺口出现日期 |
| `recovery_date` | 恢复安全线日期 |
| `evidence_snapshot` | 本次计算使用的关键证据快照 |
| `created_at` / `updated_at` | 创建和更新时间 |

建议幂等键：

```text
planned_expense_id + as_of_date + import_batch_id + rule_version
```

相同数据批次、日期和规则版本重复运行时更新原结果，不重复创建。

## 11. 老板拍板事件

只有以下结果需要创建或更新老板拍板事件：

```text
payment_assessment.decision = boss_review
```

老板拍板事件关联 `payment_assessment`，记录待选择方案和最终决定。其他付款建议不进入老板拍板事件表。

老板决定后，可根据决定生成执行任务：

```text
计划支出
→ 付款评估
→ 老板拍板事件
→ 老板决定
→ 执行任务
```

## 12. AI 按需解释

付款建议在数据更新、批次发布或每日定时计算时已经生成。页面无需等待 AI 即可展示确定性建议。

用户点击“AI 解释原因”后，将付款评估结果和允许动作传给 AI：

```json
{
  "expenseName": "投放渠道款",
  "amount": "120000.00",
  "rigidity": "deferrable",
  "decision": "defer",
  "reasonCodes": ["cash_gap_increased"],
  "gapBefore": "0.00",
  "gapAfter": "80000.00",
  "gapIncrease": "80000.00",
  "gapDate": "2026-07-25",
  "recoveryDate": "2026-08-10",
  "allowedActions": [
    "暂缓至预计回款到账后",
    "调整其他可延后支出",
    "由老板确认接受短期缺口"
  ]
}
```

Prompt 必须约束 AI：

- 不得重新计算或修改金额和日期；
- 不得增加输入中不存在的事实；
- 不得自行批准、拒绝或改变建议代码；
- 只能从 `allowedActions` 中组织可选方案；
- 输出固定结构，包括结论、原因、影响和可选动作。

AI 未配置、调用失败或输出校验失败时，使用确定性模板生成说明，不影响付款建议展示。

## 13. 运行时机

建议在以下场景运行付款建议引擎：

- 新数据批次发布成功后；
- 每日导入最新账户余额后；
- 应收预计到账日期或计划支出发生变化后；
- 规则版本升级并要求重新计算时。

每次运行必须绑定统计日期、数据批次和规则版本，保证结果可复算。

## 14. 计算示例

假设当前有三笔已审批支出：

| 支出 | 金额 | 刚性 | 计划日期 |
| --- | ---: | --- | --- |
| 工资 | 100 万 | rigid | 7 月 10 日 |
| 服务器费用 | 30 万 | rigid | 7 月 20 日 |
| 渠道投放 | 50 万 | deferrable | 7 月 8 日 |

付款队列按刚性和日期排序为：工资、服务器费用、渠道投放。

- 工资支付后不产生缺口：`pay`，加入后续曲线；
- 服务器费用支付后产生 20 万缺口：`boss_review`，作为刚性支出加入后续曲线；
- 渠道投放使缺口继续扩大 50 万：`defer`，不加入后续曲线。

首页关注顺序为：服务器费用、渠道投放、工资。

## 15. MVP 验收标准

- `cancelled` 支出不进入付款建议；
- 非 `approved` 支出不进入现金流逐笔试算；
- 已审批支出按照刚性程度和计划日期稳定排序；
- 试算前先从原预测中移除待评估支出，避免重复扣除；
- 可延后支出造成缺口时建议暂缓且不占用后续试算资金；
- 刚性支出造成缺口时升级老板拍板并占用后续试算资金；
- 同一批数据、统计日期和规则版本产生一致结果；
- Top 3 候选不足三笔时不使用模拟数据补足；
- AI 不可改变规则引擎的建议代码、金额和日期。

## 16. 待确认口径

1. `planned` 和 `pending` 支出是否继续作为现金流预测中的保留占用；
2. 刚性支出的具体业务分类由谁维护；
3. 现金安全线长期使用 0 元还是财务配置金额；
4. 计划日期相同且刚性程度相同时是否需要业务类别优先级；
5. 后续接入合同、结算单、发票和验收单时的必备资料清单；
6. 老板拍板允许的标准选项及其任务生成规则；
7. 付款建议结果保留周期和历史版本查询要求。
