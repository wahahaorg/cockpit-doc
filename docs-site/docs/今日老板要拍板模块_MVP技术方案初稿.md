# 今日老板要拍板模块 MVP 技术方案初稿

文档版本：V0.1  
状态：讨论稿，升级阈值和处理选项待产品与财务确认  
目标读者：产品、财务、后端开发、前端开发和测试人员

## 1. 模块目标

“今日老板要拍板”不是独立预测模型，也不是普通风险列表。它承接其他规则引擎中无法按既定规则直接处理、必须由老板在多个业务方案中作出选择的升级事件。

模块需要回答：

- 什么事情需要老板处理；
- 为什么升级到老板；
- 会影响多少金额、在哪一天产生影响；
- 老板可以选择哪些处理方案；
- 老板最终选择了什么；
- 决策是否已经完成。

## 2. MVP 范围

### 2.1 本期实现

- 接收回款风险和付款建议两类升级事件；
- 使用确定性规则判断是否需要老板介入；
- 保存事件来源、影响、证据、可选方案和处理状态；
- 提供“查看依据”和“AI 解释原因”；
- 提供事件类型对应的真实处理按钮；
- 保存老板选择、意见和决定时间；
- 来源风险解除时自动关闭尚未处理的事件。

### 2.2 本期不实现

- 30 / 60 / 90 天现金流缺口单独生成第三类拍板事件；
- 独立的风险责任追踪首页模块；
- “生成动作”按钮；
- “老板复盘”功能；
- 由 AI 决定是否升级或替老板作出选择；
- 自动付款、自动催收或其他不可逆业务操作。

## 3. 事件来源

MVP 只接受两个来源：

### 3.1 回款风险

```text
应收款 + 实际回款
→ 回款风险引擎
→ 命中老板升级条件
→ 老板拍板事件
```

### 3.2 付款建议

```text
计划支出 + 90 天现金流预测
→ 付款建议引擎
→ payment_assessment.decision = boss_review
→ 老板拍板事件
```

30 / 60 / 90 天现金流结果作为两类事件的计算依据和影响证据，不单独生成事件。这样可以避免同一个现金缺口同时产生“现金缺口事件”“回款事件”和“付款事件”。

## 4. 统一升级原则

只有同时满足以下含义时才进入老板拍板：

```text
存在明确业务风险
且系统无法在既定规则内直接给出可执行处理
且需要老板在多个业务方案中作出选择
```

以下情况不应升级老板：

- 系统已有明确处理规则；
- 普通责任人可以自行完成；
- 只是资料缺失或审批未完成；
- 只是展示正常、临期或轻度风险；
- AI 认为“可能重要”，但没有命中确定性升级规则。

## 5. 回款风险升级规则

满足任一条件时，回款风险进入老板拍板：

```text
逾期天数 >= 30 天
或
已经逾期且未回金额 >= 1,000,000 元
```

对应关系：

| 回款状态 | 处理方式 | 是否升级 |
| --- | --- | --- |
| 未来 2 天内临期 | 责任人正常关注 | 否 |
| 逾期 1～29 天且未回金额低于 100 万 | 责任人继续跟进 | 否 |
| 逾期达到 30 天 | 需要老板选择处理强度 | 是 |
| 已逾期且未回金额达到 100 万 | 需要老板决定是否升级催收 | 是 |
| 已全部回款 | 风险解除 | 否；已有待处理事件自动关闭 |

MVP 中红色回款风险等同于老板升级边界。阈值由规则参数管理，不得散落在页面代码中。

## 6. 付款建议升级规则

付款建议只有同时满足以下条件时进入老板拍板：

```text
付款资格检查通过
且 approval_status = approved
且 rigidity = rigid
且支付后产生或扩大现金缺口
```

对应关系：

| 付款评估结果 | 处理方式 | 是否升级 |
| --- | --- | --- |
| 资料不完整 | 补充依据 | 否 |
| 审批未完成 | 继续审批 | 否 |
| 支付后不产生或扩大缺口 | 建议可付 | 否 |
| 可延后支出导致缺口 | 建议暂缓 | 否 |
| 刚性支出导致缺口 | 系统不能直接决定不付 | 是 |
| 支出已取消 | 排除 | 否；已有待处理事件自动关闭 |

## 7. 事件生成与幂等

建议新增 `decision_event` 表保存老板拍板事件。

每个来源先生成稳定的 `business_key`：

```text
回款事件：receivable_risk:{receivable_no}
付款事件：payment_assessment:{expense_no}
```

这里使用财务模板中的稳定业务编号，而不是数据库行 UUID。每次全量导入都会生成新的数据库行 UUID，若使用 UUID，同一笔业务跨批次会被误判为新事件。

同一 `business_key` 同时最多存在一个 `pending` 事件。事件本身使用风险周期号生成 `event_key`：

```text
event_key = business_key + risk_cycle_no
```

风险持续存在期间，每日重算只更新当前事件；风险解除后再次出现，`risk_cycle_no` 加一并创建新的事件。统计日期不得直接作为风险周期标识，否则会导致同一风险每天生成一张新卡片。

当同一来源重复运行规则时：

- 已存在未处理事件：更新影响金额、日期和证据，不重复创建；
- 已存在已决定事件且出现新的独立风险周期：创建新事件；
- 来源风险已经解除：自动关闭未处理事件，并记录关闭原因；
- 规则版本变化：保留旧事件历史，新结果按新规则版本处理。

## 8. 决策事件表

`decision_event` 建议包含：

| 字段 | 含义 |
| --- | --- |
| `id` | 主键 |
| `business_key` | 来源业务键，同一来源只允许一个待处理事件 |
| `event_key` | 业务键与风险周期组成的唯一事件键 |
| `risk_cycle_no` | 同一业务来源第几次进入升级状态 |
| `event_type` | `receivable_risk` / `payment_assessment` |
| `source_id` | 来源风险或付款评估标识 |
| `source_task_id` | 可选，关联已有风险任务 |
| `title` | 老板可读标题 |
| `risk_level` | 事件风险等级 |
| `impact_amount` | 主要影响金额 |
| `impact_date` | 主要影响日期 |
| `reason_codes` | 命中的结构化升级原因 |
| `evidence_snapshot` | 生成事件时的关键计算证据 |
| `allowed_options` | 该事件允许选择的处理方案 |
| `status` | pending/decided/closed |
| `selected_option` | 老板最终选择 |
| `decision_note` | 老板意见 |
| `decided_by` | 决策人 |
| `decided_at` | 决策时间 |
| `closed_reason` | 自动或人工关闭原因 |
| `import_batch_id` | 数据批次 |
| `rule_version` | 规则版本 |
| `version` | 乐观锁版本 |
| `created_at` / `updated_at` | 创建和更新时间 |

核心判断、金额、日期、原因代码和证据快照由规则引擎写入。AI 不得修改这些字段。

## 9. 页面展示

首页只展示：

```text
status = pending
且 event_type 属于 receivable_risk / payment_assessment
```

每张事件卡至少展示：

- 事件标题和来源类型；
- 影响金额和影响日期；
- 升级原因；
- 当前责任人；
- 数据状态和计算时间；
- 查看依据；
- AI 解释原因；
- 与事件类型匹配的处理选项。

建议排序：

```text
风险等级降序
→ 影响日期升序
→ 影响金额降序
→ 事件创建时间升序
```

## 10. 真实处理按钮

原型中的“生成动作”和“老板复盘”不进入 MVP。页面必须提供真正改变事件状态的处理按钮。

### 10.1 回款风险事件

MVP 建议选项：

| 选项代码 | 页面文案 | 需要的补充输入 |
| --- | --- | --- |
| `escalate_collection` | 升级催收 | 可填写意见 |
| `continue_followup` | 继续负责人跟进 | 新截止日期 |
| `adjust_expected_date` | 调整预计到账日期 | 新预计到账日期 |
| `reassign_owner` | 调整责任人 | 新责任人 |

### 10.2 付款建议事件

MVP 建议选项：

| 选项代码 | 页面文案 | 需要的补充输入 |
| --- | --- | --- |
| `pay_and_fund` | 同意支付并安排资金 | 资金安排说明 |
| `defer_payment` | 延期支付 | 新计划日期 |
| `adjust_other_expenses` | 调整其他支出 | 可填写调整说明 |
| `prioritize_collection` | 优先催收指定回款 | 关联应收或说明 |

前端只能展示 `allowed_options` 中的选项。后端必须再次校验选项是否属于该事件，不能只依赖前端限制。

## 11. 决策提交

老板选择处理方案后：

```text
校验事件仍为 pending
→ 校验选项属于 allowed_options
→ 校验必填补充输入
→ 保存 selected_option、decision_note、decided_by、decided_at
→ status 更新为 decided
→ 根据选择更新或生成后续任务（MVP 可选）
```

使用 `version` 做乐观锁。同一个事件被两个人同时处理时，后提交者必须收到冲突提示，不得覆盖第一次决定。

如果 MVP 暂不建设任务执行闭环，至少要完整保存老板选择，不能只在页面本地改变状态。

## 12. 查看依据

“查看依据”不调用 AI，直接展示结构化证据：

- 来源应收或计划支出；
- 应收原金额、已回金额和未回金额；
- 支付前后预测余额和现金缺口；
- 缺口金额和出现日期；
- 命中的规则和规则版本；
- 数据批次及计算时间。

查看依据必须始终可用，即使 AI 未配置或调用失败。

## 13. AI 解释原因

用户点击“AI 解释原因”时再调用 AI，建议采用 SSE 流式返回：

```text
用户点击
→ 后端读取事件证据快照
→ 拼接固定 Prompt
→ AI 生成解释
→ SSE 流式返回
```

AI 可以生成：

- 一句话结论；
- 升级原因说明；
- 影响金额和日期说明；
- 对 `allowed_options` 的通俗解释。

AI 不得：

- 重新计算或修改金额和日期；
- 增加证据中不存在的事实；
- 改变事件风险等级；
- 添加 `allowed_options` 以外的处理方案；
- 代替老板提交决定。

AI 未配置、调用失败或输出校验失败时，使用固定模板解释，不影响查看依据和老板处理事件。

## 14. 事件自动关闭

每次数据更新并重新运行规则后，应检查现有 `pending` 事件：

- 应收已经全部回款：关闭回款事件；
- 应收不再命中红色升级条件：关闭回款事件；
- 付款评估不再是 `boss_review`：关闭付款事件；
- 计划支出已取消：关闭付款事件。

自动关闭时记录 `closed_reason`，不得删除事件历史。

已决定事件不因后续数据变化而改写历史决定；后续变化通过来源数据和新事件体现。

## 15. 运行时机

建议在以下场景刷新拍板事件：

- 新数据批次发布成功后；
- 每日账户余额更新并完成规则重算后；
- 应收、实际回款或预计到账日期变化后；
- 计划支出或付款评估结果变化后；
- 规则版本升级并重新计算后。

## 16. MVP 验收标准

- 只接收回款风险和付款建议两个事件来源；
- 30 / 60 / 90 天现金流缺口不重复生成独立事件；
- 黄色回款风险不进入老板拍板；
- 红色回款风险生成且只生成一个未处理事件；
- 只有刚性支出导致缺口时生成付款类拍板事件；
- 相同来源重复计算不会产生重复未处理事件；
- 来源风险解除后，未处理事件自动关闭并保留历史；
- 查看依据不依赖 AI；
- AI 只能解释，不可改变计算事实和处理选项；
- 老板决定必须持久化，并通过乐观锁避免覆盖。

## 17. 待确认口径

1. 回款升级长期采用“逾期 30 天或逾期未回 100 万元”是否合适；
2. 不同事件类型的标准处理选项和必填输入；
3. 老板决定后是否在 MVP 自动生成或更新执行任务；
4. 首页最多展示多少个待拍板事件；
5. 事件自动关闭后是否需要通知原责任人；
6. AI 解释是否缓存及缓存失效规则；
7. 拍板事件和证据快照的历史保留期限。
