# CEO 现金流驾驶舱 V3 MVP 后端主要实现说明

文档版本：V0.1  
目标读者：后端、前端、产品和后续接手该项目的工程同学  
阅读目标：读完后能判断当前 MVP 后端已经实现了哪些确定性能力、每个模块的数据来源和计算逻辑是什么，以及前端应优先对接哪些接口。

## 1. 当前后端实现总览

MVP 后端已经形成一条完整的确定性链路：

财务 Excel → 导入批次与原始行留存 → 四类标准业务数据 → 现金流 / 回款风险 / 付款建议规则计算 → 老板拍板事件沉淀 → AI 按需解释事件依据。

当前实现的核心原则是：规则引擎负责算结果，AI 只负责解释已有事实。也就是说，现金流缺口、回款风险等级、付款是否建议暂缓、是否进入老板拍板，都由后端确定性规则计算；AI 不重新算账、不编造数据、不替代老板决策。

当前主要数据源来自一个已发布导入批次。驾驶舱接口默认读取最新发布批次，未指定批次时不会混用未发布数据。

## 2. 数据导入与批次发布模块

### 2.1 模块职责

数据导入模块负责接收财务上传的标准 Excel，完成解析、字段标准化、基础校验、原始行留存，并在发布批次时切换驾驶舱当前数据版本。

当前只支持模板版本 V0.1 和 .xlsx 文件。上传文件会计算 SHA-256，重复文件会被拒绝，避免同一份源文件重复入库。

### 2.2 导入后的数据落点

一次导入会生成一个导入批次，并保留两类数据：

1. 批次级数据：文件名、模板版本、数据期间、状态、有效行数、错误行数、警告行数、发布信息和版本号。
2. 原始行数据：每个 Sheet 的原始值、标准化值、校验状态和校验消息。

原始行留存的意义是后续可以让财务或 CFO 回看“系统为什么这么算”，也方便定位 Excel 源数据问题。

### 2.3 发布逻辑

批次上传后默认进入待审核或校验失败状态。发布时会校验：

- 批次存在；
- 请求版本号与当前批次版本一致；
- 批次没有阻断错误；
- 批次状态允许发布。

发布成功后，当前批次变为最新已发布批次，原已发布批次会回到待审核状态。发布动作还会触发任务和老板拍板事件刷新。

## 3. 标准业务数据模块

### 3.1 四类核心业务数据

当前 MVP 的计算主要建立在四类标准业务数据上：

| 数据 | 作用 | 关键字段 |
| --- | --- | --- |
| 账户余额 | 作为现金流预测的期初余额来源 | 账户编码、快照日期、可用余额、受限金额、币种 |
| 应收款 | 作为预期收入、回款风险和老板拍板回款事件来源 | 应收编号、客户、应收金额、约定到账日、预计到账日、负责人、业务状态 |
| 实际回款 | 用于抵扣应收款未回金额 | 回款编号、关联应收、回款日期、回款金额、银行流水号 |
| 计划支出 | 作为现金流预测出流和付款建议来源 | 支出编号、支出名称、计划日期、计划金额、刚性程度、审批状态、负责人 |

客户信息作为应收款的基础维度单独保存；跟进记录和任务记录属于处理过程数据，不是四张基础表的一部分。

### 3.2 标准数据预览

后端已经提供账户余额、客户、应收款、实际回款、计划支出的分页预览接口。它们主要服务于后台审核、前端下钻和联调排查。

MVP 不提供标准数据的通用增删改。发现财务源数据错误时，优先修正 Excel 并重新导入，避免系统数据和财务源文件不一致。

## 4. 30 / 60 / 90 天现金流预测模块

### 4.1 模块职责

该模块生成未来 90 天逐日现金流曲线，并从同一条曲线中提取 30 天、60 天、90 天三个观察窗口的最低余额、最大缺口和缺口日期。

它是驾驶舱“30 / 60 / 90 天现金流”面板的后端来源，也是付款建议模块试算付款影响的共同公式来源。

### 4.2 数据来源

现金流预测使用以下数据：

- 期初余额：截至基准日，每个账户最新一条有效余额快照的可用余额之和；
- 预计流入：未关闭应收款的未回金额，日期取预计到账日；如果没有预计到账日，则取约定到账日；
- 预计流出：计划支出中状态为 planned、pending、approved 的有效支出；
- 预测范围：从基准日开始，连续生成 90 天。

如果某个账户在基准日前没有余额快照，会在结果中返回 warning。当前没有实际付款表，所以计划支出如果没有及时更新状态，可能造成重复预测，这一点也会返回 warning。

### 4.3 计算逻辑

每天预测余额的核心公式是：

当日预测余额 = 昨日预测余额 + 当日预计流入 - 当日计划流出。

第一天的“昨日预测余额”就是期初余额。

每一天都会计算：

- expectedInflow：当日预计到账金额；
- plannedOutflow：当日计划支出金额；
- netFlow：当日净流量；
- predictedBalance：当日预测余额。

然后分别在前 30 天、前 60 天、前 90 天窗口内计算：

- minimumBalance：窗口内最低预测余额；
- maximumGap：安全线与最低余额之间的差额；当前安全线暂定为 0；
- gapDate：最低余额出现日期；
- firstBreachDate：首次低于安全线的日期；
- recoveryDate：跌破安全线后，后续持续恢复到安全线以上的日期。

### 4.4 逾期未确认收入处理

如果一笔应收的有效预计日期早于基准日，并且仍有未回金额，它不会被强行塞进未来现金流曲线，而是计入 overdueUnconfirmedAmount。这样可以避免“已经逾期但没到账”的款项继续美化未来余额。

### 4.5 当前边界

当前预测不是机器学习预测，也不是 AI 预测；它是基于现有财务事实和计划数据的滚动测算。企业真实现金流规划通常会做 30 / 60 / 90 天视角，但应明确它是经营滚动预测，不是统计学意义上的未来预测。

## 5. 回款风险 Top 3 模块

### 5.1 模块职责

回款风险模块负责从应收款和实际回款中计算每笔应收的未回金额、逾期状态、临期状态和风险等级，并返回当前最值得关注的前三笔风险。

### 5.2 未回金额计算

每笔应收的未回金额计算方式：

未回金额 = 应收金额 - 已关联实际回款合计。

如果未回金额小于或等于 0，则该应收不再构成风险。

### 5.3 风险等级规则

当前规则版本下：

- 红色风险：逾期 30 天及以上；或者已经逾期且未回金额大于等于 100 万；
- 黄色风险：已逾期但未达到红色条件；或者距离约定到账日 0 到 2 天；
- 绿色：未逾期且不在临期范围内。

这里已经把“临期”纳入候选，只是临期默认定义为到期当天到未来 2 天。

### 5.4 排序规则

Top 3 候选只取红色和黄色风险。排序优先级为：

1. 红色优先于黄色；
2. 越接近到期或已经逾期越靠前；
3. 逾期天数越高越靠前；
4. 未回金额越大越靠前；
5. 应收编号作为稳定排序兜底。

接口会同时返回候选总数、红色数量、黄色数量和前三笔明细，方便首页展示“Top 3”之外的数量提示。

## 6. 付款建议 Top 3 模块

### 6.1 模块职责

付款建议模块负责评估计划支出能否付款、是否建议暂缓、是否需要老板介入。它不是简单按金额排序，而是把支出放入现金流曲线中逐笔试算。

评估结果会持久化到付款建议表。相同支出、相同基准日、相同导入批次、相同规则版本会幂等更新。

### 6.2 资格检查

当前 MVP 的资格检查主要基于计划支出的审批状态：

- planned、pending：未完成审批，结果为 not_ready；
- approved：进入现金流试算；
- cancelled：不作为有效付款建议。

目前资料完整性检查还没有独立字段体系，后续可以扩展为“附件是否齐全、合同是否齐全、发票是否齐全、审批链是否完整”等前置检查。

### 6.3 支付顺序

已审批支出会先排序再逐笔试算。当前排序规则：

1. 刚性支出优先于可延期支出；
2. 已经过计划付款日的支出优先；
3. 计划付款日越早越优先；
4. 支出编号作为稳定排序兜底。

这个顺序会影响每笔付款试算后的现金流缺口，因此付款建议不是彼此独立计算，而是一个队列决策。

### 6.4 决策规则

对每笔已审批支出，系统会在当前现金流曲线上模拟加入该笔付款：

- 如果加入后最大现金缺口没有增加，建议 pay；
- 如果加入后最大现金缺口增加，且该支出是刚性支出，进入 boss_review；
- 如果加入后最大现金缺口增加，且该支出是可延期支出，建议 defer。

每条建议会保留付款前后最低余额、付款前后最大缺口、缺口增加金额、缺口日期、恢复日期和证据快照。

### 6.5 Top 3 展示规则

付款建议 Top 3 的排序优先级为：

1. 需要老板拍板；
2. 建议暂缓；
3. 需要补充依据；
4. 未就绪；
5. 可付。

同一类内部再按缺口增加金额、计划日期和支出编号排序。

## 7. 今日老板要拍板模块

### 7.1 模块职责

老板拍板模块不是单独“造”任务，而是从已经计算出的高风险经营事件中抽取需要老板介入的事项，并沉淀成可决策、可留痕、可关闭的事件。

当前 MVP 事件来源只有两类：

1. 回款风险升级事件；
2. 付款建议升级事件。

现金流缺口本身目前只作为证据，不作为第三类老板事件来源。

### 7.2 回款风险进入拍板的条件

当应收款风险等级为红色时，会生成或刷新回款风险拍板事件。

事件证据包含：应收编号、客户名称、应收金额、已回金额、未回金额、约定到账日、逾期天数和命中的原因码。

老板可选处理方式当前包括：升级催收、继续跟进、调整预计到账日、重新分配负责人。

### 7.3 付款建议进入拍板的条件

当付款评估结果为 boss_review 时，会生成或刷新付款拍板事件。

典型原因是：刚性支出必须考虑支付，但支付后会扩大现金流缺口，规则引擎不能替老板选择是补资金、延期其他支出，还是强行支付。

老板可选处理方式当前包括：支付并补资金、延期付款、调整其他支出、优先催收回款。

### 7.4 事件生命周期

同一个业务事项使用稳定 business key 识别。系统会优先刷新 pending 事件，而不是重复创建多条事件。

事件状态包括：

- pending：等待老板处理；
- decided：老板已经选择处理方式；
- closed：风险源已经消失，系统自动关闭。

提交老板决策时需要带上事件版本号，避免页面打开后数据已刷新而仍提交旧决策。若选项需要补充字段，后端会做必填校验。

## 8. AI 解释模块

### 8.1 模块职责与核心原则

AI 模块只做一件事：把系统已经算好的结论，翻译成老板能读懂的中文。

**它不参与任何计算**。现金流缺口、回款风险等级、付款是否建议暂缓、是否进入老板拍板——全部由规则引擎提前算好并写入数据库。AI 只是在用户点击"AI 解释"时，把这些已有的数字和结论拼成一段通顺的说明。

```
规则引擎算数字  →  存入数据库  →  用户点击"AI 解释"  →  后端把数字传给 AI  →  AI 说人话  →  前端展示
```

### 8.2 两条解释路径

当前系统有两条 AI 解释路径，接口和行为不同：

| | 路径 A：拍板事件解释 | 路径 B：付款建议解释 |
|---|---|---|
| **触发方式** | 用户点击老板拍板事件的"查看 AI 依据" | 用户点击付款建议的"AI 解释" |
| **接口** | `GET /decision-events/{id}/ai-explanation:stream` | `POST /payment-recommendations/{id}/ai-explanation` |
| **输出方式** | SSE 流式（边生成边推送） | 同步返回 |
| **是否缓存** | 不缓存，每次都实时生成 | 缓存：首次生成后写回数据库，再次请求直接返回 |
| **缓存失效** | 不适用 | 评估结果发生变化时自动清空，下次重新生成 |

### 8.3 传入 AI 的数据格式

AI 收到的不是原始数据库记录，而是后端专门整理好的一个 JSON 事实包，只包含解释所需的字段。

#### 路径 A — 拍板事件

**回款风险事件**传入内容示例：

```json

  "eventType": "receivable_risk",
  "riskLevel": "red",
  "impactAmount": "1500000.00",
  "impactDate": "2026-06-30",
  "ownerName": "张三",
  "reasonCodes": ["overdue_30_days", "overdue_large_amount"],
  "evidence": {
    "receivableNo": "REC-2026-0088",
    "customerName": "某客户",
    "receivableAmount": "2000000.00",
    "collectedAmount": "500000.00",
    "outstandingAmount": "1500000.00",
    "agreedDueDate": "2026-05-31",
    "overdueDays": 36

  "allowedOptions": [
    "escalate_collection",
    "continue_followup",
    "adjust_expected_date",
    "reassign_owner"
  ],
  "ruleVersion": "v0.1"
}
```

**付款 boss_review 事件**传入内容示例：

```json
{
  "title": "服务器采购款付款需拍板",
  "eventType": "payment_assessment",
  "riskLevel": "red",
  "impactAmount": "800000.00",

    "expenseNo": "EXP-2026-042",

    "gapAfter": "1000000.00"

    "pay_and_fund",
    "defer_payment",
    "adjust_other_expenses",
    "prioritize_collection"
  ],
  "ruleVersion": "v0.1"
}
```

#### 路径 B — 付款建议

付款建议解释的数据结构与拍板事件略有不同，`allowedOptions` 直接是中文动作文本，并包含现金流影响的完整对比数据：

```json
{
  "expenseNo": "EXP-2026-042",
  "expenseName": "服务器采购款",
  "amount": "800000.00",
  "plannedDate": "2026-07-15",
  "rigidity": "rigid",
  "approvalStatus": "approved",
  "decision": "boss_review",
  "reasonCodes": ["rigid_expense", "cash_gap_increased"],
  "gapBefore": "200000.00",
  "gapAfter": "1000000.00",
  "gapIncrease": "800000.00",

  "recoveryDate": null,

    "支付并补充资金",
    "调整其他支出",
    "优先催收回款",
    "老板确认接受短期缺口"
  ]

```

### 8.4 Prompt 设计

两条路径各有独立的 Prompt，核心约束相同：

**系统提示词约束（两条路径共有）：**
- 只能解释传入的结构化事实，不能重新计算或修改任何金额、日期、风险等级
- 不得提出传入 `allowedOptions` 以外的处理方案
- 不得代替老板做最终决策
- 如果某项信息未提供，明确写"系统暂无该信息"，不要猜测

**输出结构要求：**

路径 A（拍板事件）：
1. 一句话结论
2. 为什么需要老板拍板
3. 对现金流或回款的影响
4. 当前可以选择的处理方式

路径 B（付款建议）：
1. 一句话结论
2. 建议原因
3. 对现金流的影响
4. 可选动作

### 8.5 降级机制

当 AI 未启用（`ENABLE_AI=false`）、未配置 API Key 或调用失败时，系统不会返回错误，而是直接用模板拼出一段确定性文本返回。

**路径 A 降级示例输出：**
```
一句话结论：该回款风险已达到老板介入条件。

为什么需要老板拍板：规则引擎命中了 2 项升级条件，现有规则无法自动代替老板选择处理方案。

影响：涉及金额 150.00 万元，影响日期为 2026-06-30。

当前可以选择的处理方式：escalate_collection、continue_followup、adjust_expected_date、reassign_owner。
```

**路径 B 降级示例输出：**
```
一句话结论：服务器采购款当前需老板拍板。

建议原因：该付款会扩大现金缺口，但属于刚性支出，系统不能自动替老板决定。

对现金流的影响：支付前最大缺口 200000.00，支付后最大缺口 1000000.00，
缺口增加 800000.00，缺口日期 2026-07-15。

可选动作：支付并补充资金、调整其他支出、优先催收回款、老板确认接受短期缺口。
```

降级文本和 AI 文本通过响应中的 `degraded: true/false` 字段区分，前端可据此决定是否展示"AI 生成"标识。

### 8.6 接入配置

在 server 的 `.env` 文件中配置：

```env
ENABLE_AI=true
AI_MODEL=qwen-plus
DASHSCOPE_API_KEY=sk-xxxxxxxxxxxx
AI_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
AI_TIMEOUT=30
AI_MAX_RETRIES=2
```

### 8.7 当前边界

- 路径 A（拍板事件）的 AI 解释不持久化，每次请求均实时生成；如需审计留痕，后续可增加解释记录表。
- 路径 B（付款建议）的 AI 解释写回 `payment_assessment.ai_explanation`，支持缓存；评估结果变化时自动失效。
- 当前不对 AI 输出做二次校验（如金额是否与传入一致），依赖 Prompt 约束控制。

## 9. 首页概览与任务模块

### 9.1 首页概览

首页概览接口会按当前月返回可用现金、预计回款、实际回款、计划支出、本月现金缺口、逾期未回金额、今日待处理任务数量和四张概览卡片。

该模块目前更偏“经营概览”，不是具体某个原型卡片的唯一数据源。驾驶舱首页中更细的模块建议优先使用现金流预测、回款风险 Top 3、付款建议 Top 3 和老板拍板事件接口。

### 9.2 任务模块

发布批次后，系统会根据逾期回款生成催收任务。红色风险任务会标记需要老板介入。

任务可以查询和更新状态，状态迁移受后端限制。完成任务不等于实际回款到账，实际到账仍必须来自回款数据。

## 12. 当前明确边界与待确认事项

以下事项目前不阻塞 MVP 前端对接，但会影响后续产品口径：

1. 安全线当前暂定为 0，后续可能需要 CFO 确认为固定金额、月支出倍数或账户维度安全线。
2. 计划支出缺少真实付款表，支出状态如果不更新，可能重复进入预测。
3. 付款资料完整性当前没有字段化，只按审批状态做前置判断。
4. 临期回款当前定义为到期当天到未来 2 天，需要产品和 CFO 确认。
5. 回款红色风险阈值当前为逾期 30 天或逾期且未回金额大于等于 100 万，需要结合企业实际确认。
6. 老板拍板事件目前只来源于回款风险和付款建议；现金缺口是否单独生成拍板事件暂不做。
7. AI 解释当前按需生成，不持久化解释文本；如需审计留痕，后续可以增加解释记录表。
8. 当前登录和权限不是重点，正式使用前仍需补认证、RBAC 和审计字段。

## 13. 后续建议

短期建议先做三件事：

1. 前端大屏优先接入已实现的四个驾驶舱模块接口，先形成一版真实数据闭环。
2. 用财务下周提供的真实收入数据跑一轮导入和发布，重点核对现金流曲线、逾期回款和付款建议。
3. 把 CFO 确认后的安全线、临期定义、红色阈值、付款优先级规则固化为下一个规则版本。

中期再考虑补齐：更灵活的数据清洗管道、真实付款事实表、AI 解释留痕、权限和审计、指标依据下钻。
