# CEO 现金流驾驶舱 V1.0 MVP 技术方案讨论稿

文档状态：讨论稿  
适用阶段：从现有前端 Demo 推进至可演示、可接入真实数据的 V1.0 MVP  
目标读者：项目负责人、AI 产品经理、财务/CFO、前端与后端研发  

## 一、文档目的

本文用于确认 CEO 现金流驾驶舱 V1.0 的产品范围、系统边界和技术实现主线。

评审完成后，各方应能够确认：

1. V1.0 要建设哪些功能；
2. 财务数据如何进入系统；
3. 指标如何计算并展示；
4. AI 问答如何查询正式经营数据；
5. 哪些输入仍需产品和财务补充；
6. 哪些能力明确不进入 V1.0。

## 二、V1.0 核心目标

V1.0 不是建设完整 ERP、财务系统或自主经营 Agent，而是跑通两个最小闭环。

### 2.1 经营数据闭环

```text
固定 Excel 模板
→ 文件上传
→ 异步解析
→ 字段校验
→ 人工核对与修正
→ 确认发布
→ 正式数据入库
→ 指标计算
→ 驾驶舱展示
```

### 2.2 AI 问答闭环

```text
用户自然语言提问
→ 大模型选择受控 Tool
→ Tool 查询正式数据或指标结果
→ 大模型基于 Tool 结果生成回答
→ 返回数据时间、来源和复核状态
→ 保存 Trace 并进行评测
```

V1.0 的最终目标是让 CEO 能够获得四类结论：

- 钱够不够；
- 风险在哪里；
- 谁在处理；
- 是否需要 CEO 介入。

## 三、V1.0 系统范围

系统由四部分组成：

1. CEO 经营驾驶舱；
2. 财务数据管理后台；
3. 指标计算与任务服务；
4. AI 问答服务。

整体关系如下：

```text
                     ┌──────────────────┐
                     │ CEO 经营驾驶舱   │
                     │ 指标/风险/待办   │
                     └────────▲─────────┘
                              │
┌──────────────┐      ┌───────┴────────┐
│ 财务管理后台 │─────▶│ 业务数据库     │
│ 上传/校验/发布│      │ 指标与任务快照 │
└──────┬───────┘      └───────▲────────┘
       │                      │
       ▼                      │
┌──────────────┐      ┌───────┴────────┐
│ 异步任务服务 │─────▶│ 规则计算服务   │
│ 解析/向量化  │      │ 聚合/预警/任务 │
└──────┬───────┘      └────────────────┘
       │
       ▼
┌──────────────┐      ┌────────────────┐
│ 文档知识库   │◀────▶│ AI 问答服务    │
│ pgvector     │      │ Tool Calling   │
└──────────────┘      └────────────────┘
```

## 四、财务数据管理后台

V1.0 需要建设轻量财务后台。即使暂不建设复杂权限体系，也不能只保留 CEO 驾驶舱，否则上传失败、数据修正、重复导入和结果复核都只能依赖研发人员处理。

### 4.1 账号与登录

V1.0 可暂时只提供一个管理员账号，具备全部后台权限。

最低要求：

- 密码加密存储；
- 登录会话和过期机制；
- 上传文件类型、大小限制；
- 基础操作日志。

复杂角色、部门权限和文件权限不进入 V1.0。

### 4.2 Excel 模板管理

V1.0 采用固定模板，不支持任意 Excel 智能识别。

第一批模板建议包括：

- 账户余额；
- 应收款；
- 实际回款；
- 计划支出；
- 预算；
- 客户与责任人。

每份模板需要明确：

- 模板名称和版本；
- Sheet 名称；
- 表头字段；
- 必填字段；
- 日期、金额和币种格式；
- 枚举值；
- 唯一标识；
- 字段说明和样例值。

模板和业务字段必须由财务提供事实依据并最终确认，由技术团队负责标准化。

### 4.3 文件上传与批次管理

上传文件后创建独立导入批次，并保存：

- 原始文件；
- 文件类型和模板版本；
- 上传时间和上传账号；
- 数据所属账期；
- 解析状态；
- 成功、失败和重复行数；
- 错误报告；
- 发布或作废状态。

原始文件保存到 MinIO，文件元数据和批次状态保存到 PostgreSQL。

### 4.4 Excel 解析与校验

解析任务异步执行，不占用上传请求。

校验至少包括：

- 文件和 Sheet 是否符合模板；
- 表头是否完整；
- 必填字段是否缺失；
- 日期、金额、币种和枚举是否合法；
- 唯一标识是否重复；
- 客户、责任人等关联对象是否存在；
- 同一文件或同一账期是否重复导入；
- 业务状态是否允许发布。

校验结果需要定位到具体 Sheet、行和字段。

### 4.5 人工核对与发布

导入数据不能立即影响驾驶舱。建议采用以下状态：

```text
已上传 → 解析中 → 校验失败/待确认 → 已发布 → 已作废
```

后台支持：

- 查看解析结果；
- 筛选错误数据；
- 在线修正或重新上传；
- 重新校验；
- 确认发布；
- 作废错误批次；
- 重新执行解析任务。

V1.0 可以由同一个管理员完成核对和发布，但必须保留发布动作和状态，不能将上传等同于正式入库。

### 4.6 数据分层

建议将数据分成三层：

```text
原始层：原始文件、原始行和导入批次
暂存层：解析完成、等待修正和确认的数据
业务层：已确认发布、可参与正式计算的数据
```

Excel 列不能直接一比一等同数据库业务表。数据库应围绕客户、合同、应收、回款、支出、预算等业务实体设计。

## 五、知识库管理

知识库只处理非结构化或半结构化文档，例如：

- 财务制度；
- 字段定义；
- 指标口径；
- 回款预警 SOP；
- 支出审批规则；
- 公司内部说明文档。

文档上传后执行：

```text
MinIO 保存原文件
→ 异步提取文本
→ 文本清洗与切片
→ Embedding
→ 写入 PostgreSQL/pgvector
```

后台最低支持：

- 文档上传；
- 解析和向量化状态；
- 失败重试；
- 启用、停用和删除；
- 查看基础元数据和切片数量。

金额、日期、应收、回款和指标结果等结构化数据不依赖向量检索，而是存入业务表并通过 Tool 查询。

## 六、指标计算与任务

### 6.1 计算原则

所有正式金额和风险结论由后端确定性规则计算。大模型不得自行计算、修改公式或根据文档推测金额。

第一批指标建议包括：

- 本月可用现金；
- 本月预计回款；
- 本月实际回款；
- 本月计划支出；
- 本月现金流缺口；
- 管理费用可用额度；
- 应收余额；
- 逾期天数；
- 回款风险等级；
- 今日待处理任务。

最终公式、阈值和数据来源须由 CFO/财务确认。

### 6.2 计算触发方式

V1.0 建议支持三种触发方式：

1. 数据批次发布后立即重算受影响指标；
2. 每日定时刷新逾期天数、风险等级和今日待办；
3. 管理员手动重新计算。

### 6.3 指标快照

指标计算结果建议形成快照，至少保存：

- 指标编码和值；
- 统计周期；
- 数据截止时间；
- 计算时间；
- 规则版本；
- 来源批次；
- 复核或发布状态。

驾驶舱和 AI Tool 优先读取正式指标快照，避免每次请求临时重复计算。

### 6.4 任务与预警

根据风险规则生成任务，记录：

- 风险类型；
- 风险等级；
- 关联客户、应收或支出；
- 责任人；
- 截止时间；
- 处理状态；
- 跟进记录；
- 是否升级。

复杂组织关系、外部消息通知和多级审批可后置，但任务生成和状态跟踪属于驾驶舱闭环的一部分。

## 七、AI 问答方案

### 7.1 V1.0 定位

V1.0 AI 问答采用轻量 LangChain 编排，不建设多 Agent、复杂规划或自主循环。

核心约束是：

> 涉及经营金额、日期、客户、风险和任务状态的回答，必须调用受控 Tool，并忠实使用 Tool 返回结果。

### 7.2 调用流程

```text
用户问题
→ LLM 识别问题和选择 Tool
→ 应用服务校验 Tool 参数和权限
→ Tool 查询业务数据库、指标快照或知识库
→ 将结构化结果返回 LLM
→ LLM 生成面向用户的回答
→ SSE 流式输出
→ 保存问题、Tool 调用、结果和最终回答
```

### 7.3 V1.0 Tool 设计

建议先提供少量边界清晰的 Tool：

```text
get_cashflow_summary(period)
查询现金、预计回款、计划支出和现金缺口

get_receivable_risks(date, risk_level)
查询逾期应收、风险等级、负责人和跟进状态

get_expense_summary(period)
查询预算、已发生金额、剩余额度和现金影响

get_today_tasks(date)
查询今日待办、责任人和处理状态

search_finance_knowledge(query)
查询制度、SOP、字段定义和指标口径
```

Tool 只读取已发布的正式数据或明确标记为可展示的数据。

### 7.4 回答规则

AI 回答至少应包含：

- 明确结论；
- 关键数字；
- 数据截止时间；
- 数据状态；
- 来源批次或来源说明；
- 必要时给出计算口径或制度依据。

以下情况必须提示或拒答：

- 没有相应数据；
- 数据尚未发布或复核；
- 问题超出 V1.0 Tool 能力；
- 用户要求 AI 修改口径或直接执行付款；
- Tool 调用失败或结果不完整。

### 7.5 RAG 与 Tool 的边界

```text
制度、定义、SOP、口径解释 → RAG/pgvector
金额、名单、日期、状态、指标 → 数据 Tool
既问数字又问原因 → 数据 Tool + 知识库 Tool
```

V1.0 不采用 Text-to-SQL。自由分析类问题可以在后续版本通过只读分析视图和受控 SQL 能力扩展。

## 八、CEO 驾驶舱与前端

前端基于现有 Demo 进行工程化和视觉优化，保留其核心信息架构。

V1.0 页面建议包括：

- 登录；
- 经营驾驶舱；
- 今日待办；
- 回款风险；
- AI 问答；
- 财务后台；
- Excel 导入与批次详情；
- 数据核对与发布；
- 知识库管理。

前端需要完成：

- TypeScript 类型定义；
- 统一 API 请求层；
- Mock 数据集中管理；
- API 契约预留；
- 加载、空数据、错误和无权限状态；
- SSE 流式对话；
- Tool 调用中的状态提示；
- 数据更新时间和复核状态展示；
- 元/万元显示切换的一致性处理。

在真实后端完成前，前端和 AI 可以使用 Mock API、Mock Tool 完成技术演示。模型调用和 Tool 路由可以是真实的，底层经营数据暂时使用固定测试数据。

## 九、技术选型建议

| 层级 | V1.0 建议 |
| --- | --- |
| 前端 | React + TypeScript |
| 后端 API | Python + FastAPI |
| 数据库 | PostgreSQL |
| ORM 与迁移 | SQLAlchemy + Alembic |
| 向量存储 | PostgreSQL + pgvector |
| 文件存储 | MinIO |
| Excel 解析 | pandas + openpyxl |
| 异步任务 | Celery + Redis |
| AI 编排 | LangChain + Function/Tool Calling |
| 流式输出 | SSE |
| 评测与 Trace | Langfuse 或 LangSmith 二选一 |
| 部署 | Docker Compose |

V1.0 采用模块化单体，不拆微服务。API 服务和 Worker 可以分别运行，但共享领域模型和数据库。

## 十、测试与 AI 评测

### 10.1 工程测试

至少覆盖：

- Excel 模板识别；
- 字段解析与校验；
- 重复导入和幂等；
- 发布、作废和重试；
- 指标公式；
- 定时刷新；
- Tool 查询；
- API；
- SSE 中断处理。

### 10.2 AI 最小评测集

V1.0 建议建立 30～50 条核心问题，先验证 Tool Calling 是否可靠。

每条用例至少包含：

- 用户问题；
- 期望 Tool；
- 期望参数；
- 标准 Tool 结果；
- 关键结论；
- 是否应拒答。

第一阶段重点指标：

- Tool 选择准确率；
- Tool 参数准确率；
- 应调用 Tool 时的调用率；
- 金额与 Tool 返回结果的一致性；
- 无数据或越界问题的拒答正确率。

金额类结果使用代码规则精确校验，不能只依赖 LLM-as-a-Judge。

## 十一、V1.0 明确不做

- 银行、ERP 或审批系统实时对接；
- 真实付款、转账或自动审批；
- 任意格式 Excel 智能识别；
- 复杂用户、部门和文件权限；
- 通用公式编辑器；
- Text-to-SQL；
- 多 Agent 和复杂 LangGraph 工作流；
- 长期记忆和自主循环；
- 多模型动态路由；
- 高并发和高可用集群；
- 全公司经营分析平台。

## 十二、需要产品和财务补充确认的输入

以下内容不确认，不影响工程框架和 Mock 演示，但会阻塞正式表结构、解析规则和真实指标计算：

1. V1.0 必须回答的核心经营问题；
2. 真实脱敏 Excel 样表；
3. 每张表和每个 Sheet 的用途；
4. 一行数据代表的业务事件；
5. 唯一标识和跨表关联方式；
6. 数据是全量覆盖还是增量追加；
7. 字段业务定义、单位和必填规则；
8. 核心指标公式和风险阈值；
9. 数据维护人和复核人；
10. 回款预警和任务升级 SOP；
11. 哪些数据可以作为正式经营结论；
12. AI 第一批标准问题和标准答案。

## 十三、建议实施顺序

### 阶段 A：契约和演示骨架

- 整理页面字段；
- 定义 API 和 Tool Schema；
- 前端工程化；
- 接入 Mock API；
- 完成真实 Tool Calling 与 SSE 演示；
- 建立最小 AI 评测集。

### 阶段 B：数据小样贯通

- 获取一份真实脱敏 Excel；
- 建立标准模板和数据字典；
- 完成上传、解析、校验、核对和发布；
- 设计第一批业务表；
- 计算一个核心指标；
- 驾驶舱和 AI Tool 读取该指标。

### 阶段 C：扩展完整 V1.0

- 补齐各类 Excel 模板；
- 完成核心指标和风险规则；
- 完成任务闭环；
- 建立知识库；
- 补齐测试、评测、日志和部署；
- 完成内测验收。

## 十四、V1.0 验收标准

V1.0 至少应满足：

- 固定 Excel 模板可以上传并异步解析；
- 错误数据可以定位、修正和重新校验；
- 数据经过确认发布后进入正式业务表；
- 核心指标由确定性规则计算并形成快照；
- 驾驶舱可以读取真实接口和正式指标；
- AI 涉及经营数据时必须调用 Tool；
- AI 回答与 Tool 返回的金额和状态一致；
- AI 能说明数据时间、来源和复核状态；
- 无数据、未发布或超范围问题能够正确提示或拒答；
- 文件、批次、计算和 AI 调用具备基础追踪能力；
- 系统可以通过 Docker Compose 部署到内测环境。
